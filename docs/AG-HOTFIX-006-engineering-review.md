# AG-HOTFIX-006 Engineering Review

## Architecture Review

The change keeps LLM behavior inside the provider layer. API routes, services, and repositories do not call Ollama directly. Prompt files remain under the prompt layer and no prompt text is introduced into Python modules.

The semantic retrieval fallback remains inside database intelligence infrastructure. The retriever receives diagnostic state from the semantic index but does not own provider-specific HTTP behavior.

## Safety Review

The hotfix does not change SQL generation or execution rules. Generated SQL still flows through the SQL validator before execution, and repository access remains the only database execution path.

## Reliability Review

Embedding cache compatibility now keys off the embedding model. This prevents stale vectors generated by one model from being reused after `OLLAMA_EMBEDDING_MODEL` changes.

Startup validation fails fast when Ollama is reachable and the configured embedding model is missing. If Ollama is unreachable during local development or offline tests, the provider logs a warning rather than blocking import-time test collection.

## Test Review

Added coverage for:

- dedicated embedding model selection
- generation model separation
- missing embedding model validation
- embedding HTTP failure diagnostics
- retriever fallback diagnostic logging

## Residual Risks

- Manual verification still requires a local Ollama installation and the `nomic-embed-text` model.
- Hash fallback remains available to preserve graceful degradation, so operational monitoring should alert on embedding fallback logs.
